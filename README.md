# Multi-Branch F&B Point-of-Sale (POS) Database Architecture

## 📌 Project Overview
This project architects a centralized, scalable relational database for a multi-branch Food & Beverage (F&B) enterprise. The schema is designed to standardize Point-of-Sale (POS) operations across multiple store locations, ensuring robust data integrity for transaction processing, inventory tracking, and staff auditing.

## 🚀 Key Objectives & Business Value
* **Multi-Store Tracking**: Engineered relational constraints linking individual transactions to specific store branches to monitor regional revenue performance.
* **Granular Transaction Auditing**: Separated `TransactionHeader` and `TransactionDetail` to support high-volume, multi-item customer orders while maintaining strict 3NF normalization.
* **Payment & Staff Accountability**: Integrated `Payment` and `Staff` entities directly into the transaction lifecycle for precise cash flow tracking and employee performance evaluation.

## 📊 Database Architecture (Entity Relationship Diagram)
The architecture consists of 8 interconnected entities designed for an enterprise retail ecosystem:

### Core Entities & Business Logic:
1. **Store**: Manages branch metadata (Location, Address) for spatial revenue analysis.
2. **Staff & Customer**: Tracks employee accountability per transaction and builds customer purchasing histories.
3. **Product & ProductType**: Hierarchical inventory categorization to streamline menu management.
4. **TransactionHeader & TransactionDetail**: The core operational hub. Links a specific `Customer`, served by a specific `Staff`, at a distinct `Store`, using a specific `Payment` method, to multiple `Product` items (QTY).

## 🛠️ Tech Stack
* **DBMS**: Microsoft SQL Server (SSMS)

## 🔍 Implementation Highlights

### 1. Complex Relational Integrity
The database enforces strict referential integrity. A transaction cannot exist without a valid Store, Staff, and Payment ID, preventing orphaned records and ensuring financial data accuracy.

### 2. Sample Analytical Query (Store Performance)
The following query demonstrates how this architecture extracts business intelligence, specifically calculating total revenue generated by each branch:

```sql
SELECT 
    s.StoreName,
    COUNT(DISTINCT th.TransactionID) AS TotalTransactions,
    SUM(td.QTY * p.ProductPrice) AS TotalRevenue
FROM TransactionHeader th
JOIN Store s ON th.StoreID = s.StoreID
JOIN TransactionDetail td ON th.TransactionID = td.TransactionID
JOIN Product p ON td.ProductID = p.ProductID
GROUP BY s.StoreName
ORDER BY TotalRevenue DESC;
